# Nomi Party Notes

## 概要

- タイトル: `深夜の廻しスマホ`
- リポジトリ: `nomi-party`
- 公開URL: `https://nomi-party.yukinooooooosan.cc/`
- 公開基盤: Cloudflare Pages
- ビルド: Vite + React
- UI基盤: React Aria Components
- Cloudflare Pages build command: `npm run build`
- Cloudflare Pages output directory: `dist`
- 位置づけ: 飲み会向けの独立パーティゲーム集

このアプリは、Folio本館とは別リポジトリで管理する。
Folio本館は入口だけを持ち、ゲーム本体はこのリポジトリで育てる。

## コンセプト

一台のスマホを順番に回して遊ぶ。
参加者全員が自分の端末を出さなくてもよく、ルール説明が短く、飲み会の途中でも始めやすいゲーム集にする。

雰囲気は、少し攻める・少し大人向け。
ただし、直接的すぎる表現や人間関係を壊しやすいお題には寄せすぎない。
笑える、少し照れる、会話が増える、くらいの温度を狙う。

## 管理方針

`nomi-party` は独立アプリとして管理する。
React/Viteを前提にし、ゲーム数、共通プレイヤー状態、ゲームごとの状態管理、専用UIが増えても耐えられる構成にする。

この判断にする理由:

- ゲーム集として画面遷移と共通状態が育ちやすい。
- 色あて、投票、秘密配布、順番制など、ゲームごとにUIや状態が大きく変わる。
- Cloudflare Pagesの設定をFolio本館と分離できる。
- 将来、DB、API、管理画面、お題管理などが必要になっても移行事故を起こしにくい。

## ゲーム追加の方針

ゲームは、それぞれ独立したフォルダとして追加する。
共通化しすぎず、ゲームごとの画面、ルール、状態、演出は各ゲームフォルダに閉じ込める。

新しいゲームを追加するときの基本形:

```text
src/games/new-game/
  index.jsx
  logic.js
  data.js
```

`logic.js` や `data.js` は必要な場合だけ置く。
お題、選択肢、画像、抽選ロジックなどがゲームごとに育ってきたら、`index.jsx` から分離する。

`index.jsx` では、そのゲームのメタデータとReactコンポーネントをexportする。

```jsx
export const newGame = {
  id: "new-game",
  number: "02",
  title: "新しいゲーム",
  summary: "ゲーム一覧に表示する説明。",
  players: "2人-",
  minutes: "5分",
  heat: 2,
  mood: "にぎやか",
  component: NewGame,
};
```

追加したゲームは `src/games/registry.js` に登録する。

```js
import { colorGuessGame } from "./color-guess/index.jsx";
import { newGame } from "./new-game/index.jsx";

export const games = [
  colorGuessGame,
  newGame,
];
```

現在登録されているゲームは `color-guess`、`desire-match`、`kink-match`。
仮ゲームを大量に並べるより、実装済みのゲームだけを登録していく。

共通化するもの:

- プレイヤー設定
- プレイヤー情報の `sessionStorage` 保存
- ゲーム一覧
- ハッシュルーティング
- ゲーム開始前にプレイヤー設定へ誘導する処理
- ゲーム画面の外枠
- Menuへ戻るボタン
- タイトル、説明、人数、目安時間、攻め度の表示
- 安全ルールの表示

共通化しすぎないもの:

- ゲームごとのルール
- ゲームごとの画面構成
- 入力UI
- 得点や判定
- ラウンド進行
- 演出
- お題データ

考え方:

```text
App側: 飲み会アプリとしての共通体験を管理する
各Game側: そのゲームだけのルールと画面を自由に持つ
```

## URLと画面遷移の方針

複数ページには分けず、1つの `index.html` の中でゲームを切り替える。
URLはハッシュで画面やゲームを表す。

現在のURL:

```text
https://nomi-party.yukinooooooosan.cc/
https://nomi-party.yukinooooooosan.cc/#setup
https://nomi-party.yukinooooooosan.cc/#menu
https://nomi-party.yukinooooooosan.cc/#color-guess
```

`/` と `#setup` はメンバー設定画面。
`#menu` はゲーム一覧。
`#color-guess` は色あてゲーム。
ゲーム途中の状態まではURLに持たせない。
途中でリロードした場合は、そのゲームの初期状態に戻る方針にする。

基本の流れ:

```text
メンバー設定
  ↓
ゲーム一覧
  ↓
各ゲーム
  ↓
ゲーム一覧
```

ゲーム直リンクにアクセスした場合も、メンバー設定がまだ終わっていなければ先にメンバー設定を表示する。
設定後は、そのまま指定されたゲームへ進む。

この方針にする理由:

- 飲み会中にスマホを回す体験を、ページ遷移で分断しない。
- ゲームごとの直接リンクは作れる。
- Cloudflare Pages側のルーティング設定を増やさなくてよい。
- 参加者という共通状態を先に作ることで、指名・順番・秘密確認が必要なゲームを後から足しやすい。

注意点:

- ハッシュはサーバーに送られないので、Cloudflare Pagesから見ると常に `/` へのアクセスになる。
- ゲームごとのSEOやSNSプレビューを個別に作る用途には弱い。
- 将来、ゲームごとの説明ページやOGPが必要になったら `/games/color-guess/` のような個別ページ化を検討する。

## プレイヤー設定

TOPでは、まず参加メンバーを作る。

- `+` ボタンでメンバーを追加できる。
- 参加人数は `2-12人` で固定する。
- 12人に達したら追加ボタンは無効化する。
- メンバー名は初期状態から `Player 1` のような仮名を表示する。
- 入力が空になった場合も、開始時または読み込み時に `Player 1` のような仮名で補完する。
- 名前入力は必須にしない。
- 各メンバーは性別情報を持つ。
- 画面上では `♂` / `♀` のトグルとして表示し、タップするたびに切り替える。

メンバー情報は同じタブ内の `sessionStorage` に保存する。
これは、その場で遊ぶための一時的な状態であり、サーバーやDBには保存しない。
ブラウザを閉じたり別タブで開いた場合は、再設定する前提にする。

性別情報は後続のゲームで、お題の出し分け、指名候補、秘密情報の割り当てなどに使える共通データとして扱う。

## 共通プレイヤーカラー

参加者を視覚的に追いやすくするため、プレイヤーごとの固定色を共通要素として扱う。
ルーレット、リストソート、強制ペアなど、同じ参加者が複数UIに出るゲームでは、同じ `player.id` に同じ色を使う。

### 色の優先順位

`深夜の廻しスマホ` のベースUIカラーは、青系の夜色と金色を基本にする。
これはアプリ全体の世界観、通常カード、見出し、メニュー、共通操作のための色。

一方で、プレイヤーカラーは「誰の番か」「誰に渡すか」「誰の結果か」を示す機能色として扱う。
特定プレイヤーに紐づく画面やカードでは、ベースUIカラーよりプレイヤーカラーを優先する。

考え方:

- ベースUIカラー: アプリの雰囲気、夜、金、通常状態、共通操作を担う。
- プレイヤーカラー: 操作主体、手渡し先、結果に表示された本人を示す。
- 通常画面は青系金でまとめる。
- 誰かに紐づいた瞬間だけ、その人の担当色を前に出す。
- プレイヤーカラーは装飾色ではなく、スマホを回す時の事故防止と認識補助のための色。

基本方針:

- ページまたはゲームに入った時点で、`player.id` ごとに色を固定する。
- `全員`、`♂のみ`、`♀のみ` の対象切り替えで色を再割り当てしない。
- シャッフルやペア組み替えでも、名前と色の対応は変えない。
- 見た目の調和より、誰がどの色か一目で分かる識別性を優先する。
- 参加者上限はシステム上12人だが、将来の共通部品として `男12色`、`女12色` の計24色を用意しておく。
- 性別ごとの雰囲気は少し残すが、色が近すぎると本末転倒なので、男女どちらも色相を広く散らす。

- 新規プレイヤーが増えた場合は、未使用色を優先して補完する。
- 24色を超えた場合は循環してよいが、通常の上限では発生しない想定。

### 手渡し画面での使い方

スマホを「次の人に渡す」タイプの画面では、次に操作するプレイヤーの固定色をテーマカラーとして使う。
このアプリでは、色は単なる装飾ではなく「いま誰の番か」「誰に渡すべきか」を示す共通UI言語として扱う。

対象例:

- `次の入力者`
- `次の回答者`
- `次の投票者`
- 親から子へ渡す画面
- 秘密入力や秘密回答の直前画面
- 本人だけが操作する `PRIVATE TURN` / `PRIVATE VOTE` 画面
- 結果発表で特定プレイヤーを表示するカード

使い方の方針:

- 手渡しカードは、渡し先プレイヤーの色を背景の主色にする。
- 本人が操作するプライベート画面は、操作中プレイヤーの色を背景の主色にする。
- プライベート画面の確定ボタンも、操作中プレイヤーの色に合わせる。
- 白系や明色プレイヤーでも読めるように、文字色は `getPlayerTextColor(player)` で背景色に合わせる。
- 斜めグラデーションを強く乗せるとプレイヤー色が分かりにくくなるため、プレイヤーカラーを見せたいカードは単色ベースにし、必要な場合だけ薄い明暗レイヤーを重ねる。
- 複数人が同時に出る結果画面では、それぞれの固定色を個別に使い、単一テーマカラーには寄せない。
- 発表後に特定プレイヤーが明らかになるカードは、その瞬間から該当プレイヤーの色へ切り替える。
- 色はゲームごとに再生成せず、共通の `player.id -> color` マップを参照する。

ゲーム固有の色が別に存在する場合:

- `color-guess` の「親が作った色」「回答者が予想した色」のようなゲーム内入力色は、プレイヤーカラーとは別物として扱う。
- 結果リストの行背景は回答者のプレイヤーカラーにする。
- 予想色や親の作成色は、HEX表記の横に丸いスウォッチとして表示する。
- 「誰の結果か」はプレイヤーカラー、「どんな色を選んだか」はスウォッチ、という役割分担にする。

実装上の注意:

- CSS変数は、その変数を使う要素自身または祖先に置く。兄弟要素には継承されない。
- 例: `PRIVATE TURN` カードだけに `--private-color` を指定しても、外側の確定ボタンには届かない。
- カードとボタンの両方で同じ色を使う場合は、共通の親要素に変数を置くか、両方に同じstyle変数を渡す。
- `--handoff-color` / `--handoff-text-color` は手渡しカード用。
- `--private-color` / `--private-text-color` は本人操作画面用。
- 結果表示で個別カードに色を付ける場合は、そのカード単位で専用変数を渡す。

## 現在のゲーム

### 色あて

親が作った「彼女の色」を、子が直感で当てるゲーム。
現在はRGBスライダーで色を作る。
入力中はHEX値を表示しない。
結果画面では答え合わせ用に、親の色と回答色のHEX値を表示する。
RGBスライダーはReact Aria ComponentsのSliderを使う。
見た目はこのアプリ側のCSSで整え、各バーの背景は固定の黒から原色へのグラデーションにする。
つまみだけ現在値の色を反映する。

基本フロー:

```text
親をランダム選出
  ↓
親がRGBで色を作る
  ↓
子が順番に親の色を想像してRGBで回答する
  ↓
RGB距離で近さを計算
  ↓
100点満点のスコアと順位を表示する
```

親は `♀` プレイヤーから優先してランダム選出する。
`♀` がいない場合は全プレイヤーから選ぶ。
呼び名はゲーム中の受け渡しでは `くん` / `ちゃん` を付ける。
結果カード上の名前表示では素のプレイヤー名を使う。
ランダム結果文言内では `くん` / `ちゃん` を付ける。

結果画面では親の色を大きな画像カードで見せる。
子の回答は色丸ではなく、回答色そのものをカード背景にする。
親カードは補色ベースのグラデーション背景とアニメーションを持つ。
親名をタップすると `pan2.png` / `pan2skin.png` を切り替える。
親画像をタップするとハート演出を出す。

ファイル:

```text
src/games/color-guess/
  index.jsx
  PantyPreview.jsx
  color.js
```

役割:

- `index.jsx`: 色あてゲーム本体、メタデータ、UI、状態管理
- `PantyPreview.jsx`: 色付き画像プレビューと結果用画像表示
- `color.js`: RGBからHEXへの変換、RGB距離、100点スコア計算

今後の拡張候補:

- プレイヤーごとの回答履歴
- 参加者ごとの投票
- 結果文言のバリエーション追加

### 匿名おねだりマッチング

実装IDは `desire-match`。
ゲーム一覧上のタイトルは `欲望ガチャ`。
「内容の同意が先、人間のマッチングが後」という思想の、王様ゲーム風マッチングゲーム。
少し攻めた雰囲気を持たせつつ、女性側が事前に「これならOK」を選べる設計にする。

基本思想:

- 男性側は匿名で欲望を書く。
- 女性側は欲望主を知らない状態で、応じてもよい欲望だけに投票する。
- 抽選では、先に実施する欲望をランダムに選ぶ。
- 次に、その欲望に応じてくれる女性をワンショットで発表する。
- 最後に、その欲望主を発表する。
- 投票したが当選しなかった女性の名前は表示しない。

参加条件:

- 既存のプレイヤー設定を使う。
- `♂` が欲望入力側。
- `♀` が投票側。
- 男性は1人1つだけ欲望を入力する。
- 女性は複数の欲望に投票できる。

画面フロー:

```text
ゲーム開始
  ↓
男性入力ターン
  男性が順番にスマホを持つ
  1人1つだけ欲望を自由入力する
  確定後は入力内容を隠し、次の男性へ渡す
  ↓
女性投票ターン
  女性が順番にスマホを持つ
  欲望リストから、応じてもよいものにチェックする
  複数選択可
  確定後はチェック状態をリセットし、次の女性へ渡す
  ↓
欲望抽選ターン
  欲望リストを表示する
  欲望主は表示しない
  「実施する欲望を抽選」で、未抽選の欲望から1つをランダム選出する
  ↓
実施欲望発表
  選ばれた欲望だけを表示する
  「応じてくれる女性を抽選」を押す
  候補女性リストは表示しない
  ↓
結果発表
  欲望発表
  これに応じてくれる女性
  その欲望主
  ↓
次の欲望抽選へ
```

抽選ロジック:

```text
実施欲望候補:
  まだ抽選されていない欲望

応じてくれる女性候補:
  その欲望に投票した女性
  かつ
  まだ当選していない女性
```

候補女性がいる場合は、その中からランダムに1人を選ぶ。
選ばれた女性は当選済みリストに入り、以降の抽選候補から除外する。

候補女性がいない場合は `不成立` とする。
不成立でも、その欲望は抽選済みとして扱う。
不成立の場合、当選済み女性は増えない。

結果発表の順番:

```text
欲望発表
「〇〇してほしい」

これに応じてくれる女性
△△ちゃん

欲望主
□□くん
```

不成立の場合:

```text
欲望発表
「〇〇してほしい」

これに応じてくれる女性
今回はいません

欲望主
□□くん
```

安全設計:

- 欲望入力前に、相手が断れる内容だけを書くよう短く示す。
- 女性投票画面では、少しでも迷うものは選ばなくてよいことを示す。
- 結果発表後もパスできる前提を画面に出す。
- 候補者リストは表示せず、当選者だけをワンショットで出す。
- 誰がどの欲望に投票したかは表示しない。

実装IDは `kink-match`。
ゲーム一覧上のタイトルは `性癖マッチング`。
全員が2択質問に答え、その回答ベクトルから相性を計算する診断型ゲーム。
`欲望ガチャ` が実行・抽選型なのに対して、こちらは全員回答後にペア相性をランキング化する。

基本方針:

- 質問はすべて2択にする。
- スキップは作らない。
- 回答は `left = -1`、`right = +1` として扱う。
- 相性は `-100%` から `+100%` で表示する。
- `+100%` は完全に噛み合う、`-100%` は完全に噛み合わない、という見せ方にする。

質問タイプ:

```js
questions = [
  {
    id: "praise",
    type: "reverse",
    left: "褒めたい",
    right: "褒められたい",
  },
  {
    id: "pace",
    type: "straight",
    left: "ノリ派",
    right: "じっくり派",
  },
];
```

`reverse` は、異性ペアでは逆の回答が噛み合う質問。
例: `褒めたい` と `褒められたい`、`リードしたい` と `リードされたい`。

`straight` は、同じ回答同士が噛み合う質問。
例: `ノリ派` 同士、`じっくり派` 同士。

ペアルール:

```text
opposite-only:
  異性ペアのみを候補にする

allow-same:
  異性ペアも同性ペアも候補にする
```

重要な計算ルール:

`allow-same` では、同じランキング計算の中に、`reverse` を反転するペアと反転しないペアが混在する。

```text
♂ × ♀:
  reverse質問は反転して計算する
  逆の回答が高評価になる

♂ × ♂:
  reverse質問も反転しない
  同じ回答が高評価になる

♀ × ♀:
  reverse質問も反転しない
  同じ回答が高評価になる
```

つまり、`question.type === "reverse"` だけで一律に反転してはいけない。
ペアルールと、比較している2人の性別組み合わせを見て反転するか決める。

想定ロジック:

```js
function isPairAllowed(playerA, playerB, pairRule) {
  if (pairRule === "opposite-only") {
    return playerA.gender !== playerB.gender;
  }

  if (pairRule === "allow-same") {
    return true;
  }

  return true;
}

function shouldInvertForQuestion(question, playerA, playerB, pairRule) {
  if (question.type !== "reverse") return false;

  if (pairRule === "opposite-only") {
    return true;
  }

  if (pairRule === "allow-same") {
    return playerA.gender !== playerB.gender;
  }

  return false;
}
```

相性計算:

```js
function buildVector(answer, questions) {
  return questions.map((question) => answer[question.id]);
}

function buildMatchVector(answer, questions, playerA, playerB, pairRule) {
  return questions.map((question) => {
    const value = answer[question.id];
    return shouldInvertForQuestion(question, playerA, playerB, pairRule)
      ? -value
      : value;
  });
}

score = cosineSimilarity(vectorA, matchVectorB);
percent = Math.round(score * 100);
```

`score` は `-1` から `+1`。
表示は `percent` として `-100%` から `+100%` にする。

質問数とグループ:

- 1回のプレイで出題する質問数は、開始時に `10問` / `20問` から選ぶ。
- まずは1人ずつ、選択した質問数ぶんを連続回答する。
- 質問グループを複数用意する。
- 各グループには最低 `20問` ずつ質問を用意する。
- ゲーム開始時に、各グループをどれくらい混ぜるか選べるようにする。
- 選ばれた混ぜ具合に応じて各グループから質問を抽出し、合計質問数にする。
- 抽出後、質問全体をシャッフルして出題する。

配分ルール:

```text
1. disable以外のグループのweight合計を出す
2. 選択した質問数をweight比で各グループへ配分する
3. 端数は小数部分が大きいグループから順に追加する
4. 各グループから配分数ぶんランダム抽出する
5. 合計質問数になったら、全体をシャッフルする
```

20問の場合の例:

```text
普通        たくさん  weight 4
恋愛        中くらい  weight 2
ちょい攻め  少し      weight 1
アダルト    disable   weight 0
性癖        disable   weight 0

合計weight 7

普通        約11問
恋愛        約6問
ちょい攻め  約3問
合計20問
```

注意点:

- 全グループが `disable` の場合は開始できない。
- もしくは `普通: たくさん` に自動補正する。
- グループ内の質問数が配分数に足りない場合は、そのグループの全問を使い、余りを他グループへ再配分する。
- アダルトや性癖グループは、場の空気に合わせて明示的に有効化する設計にする。

ランキング:

全プレイヤーの回答後、候補ペアを総当たりで作り、相性を計算する。
その後、スコア降順に並べる。

```text
Player 1 × Player 4  +92%
Player 2 × Player 5  +61%
Player 3 × Player 6  -12%
```

ただし、最終表示は単純なランキングではなく、上位からペアを確定して、使われた人を抜いていく方式にする。
同じ人が何度も出ないようにするため。
結果は確定したペアをすべて対象にするが、画面では1組ずつ発表する。
一気に全ペアを出さず、`次のペアを見る` のような導線で順番に開示する。
奇数人数や `opposite-only` の男女比によって組み合わせが成立しなかった参加者は、余りとして表示する。
文言は `今回は残念ですが...` のように、素直だが強すぎない表現にする。

同点の扱い:

- 同じスコアのペア同士はシャッフルしてから greedy matching に流す。
- 同点時に入力順やプレイヤー順が結果に出ないようにする。

スコアコメント:

スコア帯ごとにコメントを出す。
プラスだけでなくマイナスにもコメントを持たせる。

例:

```text
+80% 以上:
  かなり噛み合う

+40% 以上:
  いい感じに合う

0% 付近:
  まだ読めない

-40% 以下:
  ちょっとズレてる

-80% 以下:
  犬猿の仲
```

```js
function buildGreedyMatches(pairRankings) {
  const usedPlayerIds = new Set();
  const matches = [];

  for (const pair of pairRankings) {
    const [a, b] = pair.players;

    if (usedPlayerIds.has(a.id) || usedPlayerIds.has(b.id)) {
      continue;
    }

    matches.push(pair);
    usedPlayerIds.add(a.id);
    usedPlayerIds.add(b.id);
  }

  return matches;
}
```

注意点:

- greedy matching は全体最適ではない。
- ただし飲み会ゲームとしては「上位から抜けていく」説明が直感的なので採用候補にする。
- 奇数人数の場合は1人余る。
- `opposite-only` では男女数の偏りによって余りが出る。
- マイナス相性のペアも表示対象にする。マイナスもこのゲームの面白さとして扱う。
